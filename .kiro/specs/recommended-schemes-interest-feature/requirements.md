# Requirements Document

## Introduction

This feature adds "I am interested" functionality to the recommended schemes displayed on the home page. Currently, this functionality exists only on the schemes page cards. Users will be able to mark their interest in recommended schemes directly from the home page by clicking a "View Details" button, which will open a dialog with scheme information and an "Apply Now" button that triggers the interest confirmation dialog.

The implementation will replicate the existing behavior from the schemes page, ensuring consistency across the application.

## Glossary

- **Recommended_Schemes**: Personalized scheme recommendations displayed on the home page, generated by the AI recommendation system based on user profile
- **Interest_Dialog**: A modal dialog that asks users "Are you interested in this scheme?" when they click "Apply Now"
- **Scheme_Detail_Dialog**: A modal dialog that displays full scheme information including name, description, category, level, eligibility explanation, and action buttons
- **PersonalizedResultsDisplay**: The React component that renders recommended scheme cards on the home page
- **Home_Page**: The landing page of the application that displays personalized recommendations for registered users
- **Schemes_Page**: The page that displays all available schemes with search and filter capabilities
- **Interest_Service**: The backend API service that stores user interest in schemes at endpoint `/api/v1/interested-schemes`

## Requirements

### Requirement 1: Add View Details Button to Recommended Scheme Cards

**User Story:** As a user viewing recommended schemes on the home page, I want to see a "View Details" button on each scheme card, so that I can access detailed information about the scheme.

#### Acceptance Criteria

1. THE PersonalizedResultsDisplay SHALL display a "View Details" button on each recommended scheme card
2. WHEN a recommended scheme has an apply_link, THE PersonalizedResultsDisplay SHALL display both "View Details" and "Apply Now" buttons
3. WHEN a recommended scheme does not have an apply_link, THE PersonalizedResultsDisplay SHALL display only the "View Details" button
4. THE "View Details" button SHALL use the same styling and positioning as the button on Schemes_Page cards
5. THE "View Details" button SHALL include an appropriate ARIA label for accessibility

### Requirement 2: Open Scheme Detail Dialog from Recommended Schemes

**User Story:** As a user viewing recommended schemes on the home page, I want to click "View Details" to see full scheme information, so that I can make an informed decision about my interest.

#### Acceptance Criteria

1. WHEN a user clicks "View Details" on a recommended scheme card, THE Home_Page SHALL open the Scheme_Detail_Dialog
2. THE Scheme_Detail_Dialog SHALL display the scheme's official name, description, category, level, and ministry information
3. THE Scheme_Detail_Dialog SHALL display eligibility explanation from the AI recommendation
4. THE Scheme_Detail_Dialog SHALL display "Apply Now" and "Close" action buttons
5. WHEN the dialog is open, THE System SHALL trap keyboard focus within the dialog for accessibility
6. WHEN the dialog closes, THE System SHALL return focus to the element that triggered the dialog

### Requirement 3: Trigger Interest Dialog from Recommended Schemes

**User Story:** As a user viewing a recommended scheme's details, I want to be asked if I'm interested when I click "Apply Now", so that the system can save my preferences for future reference.

#### Acceptance Criteria

1. WHEN a user clicks "Apply Now" in the Scheme_Detail_Dialog opened from a recommended scheme, THE System SHALL display the Interest_Dialog
2. THE Interest_Dialog SHALL display the text "Are you interested in this scheme?"
3. THE Interest_Dialog SHALL provide "Yes" and "No" action buttons
4. WHEN a user clicks "Yes", THE System SHALL call the handleMarkInterested function
5. WHEN a user clicks "No", THE System SHALL close both dialogs without saving interest
6. THE Interest_Dialog SHALL remain open until the user makes a selection

### Requirement 4: Save User Interest for Recommended Schemes

**User Story:** As a user who confirms interest in a recommended scheme, I want my interest to be saved to the system, so that I can access it later in the Education page.

#### Acceptance Criteria

1. WHEN a user confirms interest by clicking "Yes" in the Interest_Dialog, THE System SHALL send a POST request to the Interest_Service
2. THE POST request SHALL include profile_id, scheme_name, scheme_slug, scheme_description, scheme_benefits, scheme_ministry, and scheme_apply_link
3. WHEN the Interest_Service successfully saves the interest, THE System SHALL display a success message
4. WHEN the Interest_Service fails to save the interest, THE System SHALL log an error message
5. THE System SHALL close both dialogs after the save operation completes
6. THE saved interest SHALL be retrievable from the Interest_Service by profile_id

### Requirement 5: Maintain Consistency with Schemes Page Behavior

**User Story:** As a user familiar with the schemes page, I want the "I am interested" feature on recommended schemes to work identically, so that I have a consistent experience across the application.

#### Acceptance Criteria

1. THE Interest_Dialog triggered from recommended schemes SHALL use the same component as the Schemes_Page
2. THE Scheme_Detail_Dialog opened from recommended schemes SHALL use the same component as the Schemes_Page
3. THE handleMarkInterested function SHALL use the same implementation for both recommended schemes and regular schemes
4. THE success and error messages SHALL be identical for both recommended schemes and regular schemes
5. THE dialog behavior, styling, and animations SHALL be identical for both recommended schemes and regular schemes

### Requirement 6: Handle Missing Profile ID

**User Story:** As a system, I want to handle cases where a user attempts to mark interest without a valid profile, so that the application doesn't crash or behave unexpectedly.

#### Acceptance Criteria

1. WHEN a user clicks "Apply Now" without a valid profile_id in localStorage, THE System SHALL log an error message
2. WHEN a user clicks "Apply Now" without a valid profile_id prop, THE System SHALL check localStorage for profile_id
3. IF no profile_id is found in either location, THEN THE System SHALL close the dialog without showing the Interest_Dialog
4. THE System SHALL not attempt to call the Interest_Service without a valid profile_id
5. THE error handling SHALL prevent application crashes or undefined behavior

### Requirement 7: Transform Recommended Scheme Data for Interest Service

**User Story:** As a developer, I want to correctly map recommended scheme data to the Interest_Service API format, so that all required fields are properly populated.

#### Acceptance Criteria

1. THE System SHALL extract scheme_name from the PersonalizedScheme name field
2. THE System SHALL extract scheme_slug from the PersonalizedScheme slug field
3. THE System SHALL extract scheme_description from the PersonalizedScheme description field
4. THE System SHALL extract scheme_benefits from the PersonalizedScheme benefits_summary or benefits field
5. THE System SHALL extract scheme_ministry from the PersonalizedScheme ministry field
6. THE System SHALL extract scheme_apply_link from the PersonalizedScheme apply_link field
7. WHEN a required field is missing, THE System SHALL provide an empty string as the default value

### Requirement 8: Accessibility Compliance for New Interactions

**User Story:** As a user who relies on assistive technologies, I want the new "View Details" functionality to be fully accessible, so that I can navigate and interact with recommended schemes independently.

#### Acceptance Criteria

1. THE "View Details" button SHALL have a descriptive ARIA label that includes the scheme name
2. THE Scheme_Detail_Dialog SHALL have an appropriate role attribute set to "dialog"
3. THE Scheme_Detail_Dialog SHALL have an aria-labelledby attribute pointing to the dialog title
4. THE Interest_Dialog SHALL have an appropriate role attribute set to "dialog"
5. THE Interest_Dialog SHALL have an aria-labelledby attribute pointing to the dialog title
6. WHEN a dialog opens, THE System SHALL announce the dialog content to screen readers
7. THE keyboard navigation SHALL allow users to tab through all interactive elements in the dialogs
